# План: интеграция 10 досье по колену в схему Azarean Rehab

> Все числа ниже посчитаны кодом по извлечениям: **10 досье, 44 строки фаз** (основные траектории), **154 строки `transition_criteria`**, **67 уникальных `measurement_type`** в критериях. Где досье явно требуют второй траектории (PCL, ACL) — учтено отдельно.

### Где одна патология требует ДВУХ program_type
- **ПКС:** `knee_acl` (хирург, эталон) + `knee_acl_conservative` (copers) — разные системы гейтов, нет постоп-guardrail.
- **ЗКС:** `knee_pcl_postop` + `knee_pcl_conservative` — досье явно (раздел 13 Recommendations #1): разные триггеры старта, гейты, брейс-режимы; объединение приведёт к ошибочному переносу рестрикций.

### Где НЕ дробить (один path + модификатор-флаг)
- **EMR:** PTR vs QTR — модификатор темпа, не отдельный path. Консерватив (частичные разрывы, Popkin-Golman) = тот же путь без защиты шва.
- **ITBS:** бегун vs велосипедист — поле «активность», не отдельный path.
- **PFPS:** подросток/бегун/PFOA/сидячий — слой модификаторов.
- **Patellar tendinopathy:** quadriceps-вариант + континуум reactive/degenerative — модификаторы.
- **TKA:** UKA — флаг-модификатор (смягчает ранние пороги).
- **Osteotomy:** HTO (база) / DFO / сочетанные процедуры — наложенные модификаторы (доминирует рестриктивный).

**Итого program_types: 12** (1 расширение `acl` + 11 новых). При желании сократить — 2 консервативных ПКС/ПКС можно отложить (см. §7).

**`rehab_phases` (контент «Путь»):** ~**44 строки фаз** (основные траектории). Все поля досье ложатся на существующие колонки 1:1:
- `goals` → JSON array `goals` ✓
- `restrictions` → JSON array `restrictions` ✓
- `allowed` → JSON array `allowed` ✓
- `pain_guidance` → JSON array `pain` ✓
- `red_flags` → JSON array `red_flags` ✓
- `title_ru` → `title`, `duration_weeks` (VARCHAR-диапазон "0-2"/"36+") ✓ — формат досье уже совпадает.

## 4. Что НЕ ложится в текущую модель (структурные находки)

**1. Criterion-driven vs time-driven — главный конфликт.** Существующая модель `acl` привязывает фазы к `surgery_date` (недели от операции). Но 5 из 10 досье — **`criterion_driven`** (ITBS, PFPS, тендинопатия, OA; ACL/PCL/EMR/остеотомия/TKA сами называют недели «guardrail, не гейт»). Для них `duration_weeks` либо пустой, либо ориентир от `treatment_start`, а не от операции.
→ *Схема:* `rehab_phases.duration_weeks` уже VARCHAR и допускает "" — ОК. Но **anchor нужно хранить на уровне program_type** (нет такого поля). *UI:* «Путь» не должен показывать «неделя X от операции» для не-хирургических — нужен признак anchor (surgery_date / treatment_start / criterion_only).

**2. Фаза 0 = ДО операции (ACL prehab, TKA prehab).** Отрицательное смещение от `surgery_date`. Модель «недели от операции» ломается на отрицательном времени.
→ Хранить Фазу 0 без duration_weeks, гейт = «операция выполнена» (instructor_check), переход по факту surgery_date.

**3. Двунаправленные переходы (шаг НАЗАД).** ITBS и тендинопатия: усиление боли через 24ч → откат на предыдущую фазу. OA: бессрочное поддержание + откат при flare.
→ Текущая `phase_stuck_alerts` — только «застрял». Нужна поддержка регресса фазы (или явный «откат» как мониторинг-триггер). UI «Путь» сейчас предполагает монотонное движение вперёд.

**4. Бессрочные/безвыходные фазы.** Тендинопатия стадия 5 «Поддержание» и OA этап 2 «Поддержание» (12 нед → бессрочно, `transition_criteria = []`). OA этап 3 — это «выход к врачу», не реабилитация.
→ Фазы с пустыми критериями валидны в схеме; UI должен корректно рисовать «финальную/бессрочную» фазу без кнопки «дальше».

**5. Load-based прогрессия (тендинопатия).** Дозы заданы параметрами нагрузки (RM, %MVIC, 5×45с, «каждый 3-й день» по 72-ч коллагеновому ответу), а не неделями. Это контент упражнений (комплексы/ARC-CYCLE блоки), не фазы.
→ Не ложится в `rehab_phases`; это связка с комплексами/`program_blocks`.

**6. Очень короткий курс (ITBS): 44% разрешаются к 8 нед.** Горизонт несопоставим с ACL (9-12 мес).
→ Схеме всё равно (VARCHAR), но UI-таймлайн «Путь» не должен растягивать 3-фазный 6-недельный путь по той же визуальной сетке, что 7-фазный годовой ACL.

**7. Ветвление по тяжести/локализации (хрящ).** Досье рекомендует **6 program_type** (процедура × локализация: microfracture/ACI/OAT × condyle/PF), т.к. WB/ROM-логика переворачивается (PF: нагрузка в разгибании рано, флексия поздно; condyle: разгрузка рано, ROM рано), а сроки расходятся в 2-3 раза.
→ *Решение:* засеять `knee_cartilage_repair` по ОСНОВНОЙ траектории (ACI/MACI мыщелок, 4 фазы), процедуру/локализацию вынести в модификатор-флаг. Полное дробление на 6 — отложить (см. §7).

**8. Surgeon-overridable числа.** ACL, PCL, EMR, TKA, остеотомия, хрящ: ранние WB/ROM/брейс/сроки задаёт хирург. Досье требуют «surgeon-overridable флаги», а не жёсткие правила.
→ Нет механизма «переопределить порог фазы для конкретного пациента». Сейчас критерии глобальны на `phase_id`. Это per-patient override — отдельная работа (отложить).

**9. Weekly guardrail (safety-net).** TKA (<70-80° на 2 нед / <90° на 4-6 нед → MUA-окно), тендинопатия/OA (боль не нарастает неделя-к-неделе). Это **мониторинг-триггер, НЕ критерий перехода** — досье прямо это разделяют (аналог `phase_stuck_alerts`).
→ Реализовать как алерты, не как `phase_transition_criteria`.

## 5. Слоты работы A–D

### Слот A — новые program_types (данные, `program_types` seed)
**Объём:** 11 INSERT (+1 UPDATE на `acl` если расширяем до 7 фаз). Тривиально (миграция-seed, идемпотентно `ON CONFLICT (code) DO NOTHING`).
**Зависимости:** нет. **Но** все anchor ≠ surgery_date упираются на проблему «у program_types нет колонки anchor» (§4 п.1) → либо отложить семантику anchor в код фронта, либо добавить `program_types.anchor VARCHAR` (микро-миграция, Слот D).
**Оценка:** S (полдня на seed + тест-цикл идемпотентности).

### Слот B — контент фаз (`rehab_phases` seed)
**Объём:** ~**44 строки** (или меньше, если пилотим подмножество). Поля 1:1 ложатся (§2). Расширение `acl` +2 фазы (0 prehab, 6 профилактика) — отдельный аккуратный INSERT, чтобы не ломать живой прод-ACL.
**Зависимости:** Слот A (program_type FK). Иконки/цвета — косметика, не блокер.
**Риск:** редактирование/нормализация русских текстов из JSON (часть полей пустые — заполнить плейсхолдером или оставить []). LF/UTF8 в миграции (известная грабля проекта).
**Оценка:** M (объём текста большой, но механический; основное время — вычитка клинических формулировок).

### Слот C — структурированные критерии (`phase_transition_criteria` seed)
**Объём:** ~**154 строки**. Поля ложатся (§2). ~60% — `measurement` с порогом, ~40% — instructor_check/self_report без числа.
**Зависимости:** Слот B (FK `phase_id` → `rehab_phases(id)` — нужны реальные id после seed фаз → seed критериев вторым шагом / через подзапрос JOIN на (program_type, phase_number), как сделано для ACL-seed `20260518`).
**Нормализация measurement_type ОБЯЗАТЕЛЬНА перед seed:** свести синонимы боли (`vas_pain`/`vas_pain_nprs`/`nprs_pain`/`vas_pain_sldecline_squat` → один код `vas_score`), договориться о неймспейсе LSI-кодов. Иначе словарь measurement_type расползётся на 67 несогласованных строк.
**Оценка:** M (seed механический, но нормализация словаря — это решение, требующее §6).

### Слот D — код/UI
Три подзадачи разной тяжести:

**D1. Новые measurement-хранилища (сила LSI%, hop%, баланс sec, PROMs-баллы).** Самый дорогой кусок. Сейчас система хранит ROM/girth/pain (Волна 2). Чтобы силовые/hop/баланс/PROM-критерии авто-проверялись, нужна новая таблица замеров (по образцу `rom_measurements`: type/value/unit/measured_by/session_id) + ввод инструктором + API + UI. **Без этого ~30+ критериев навсегда instructor_check.**
**Оценка:** L (новая Волна, по масштабу ~Волна 2 measurements).

**D2. Вывод авто-критериев в «Путь» (RoadmapScreen).** Сейчас «Путь» рендерит `criteria_next` как **простой текст**; структурированные `phase_transition_criteria` с галочками «выполнено/не выполнено» **не выведены**. Это backend-агрегат (сравнить последний замер vs threshold с учётом `staleness_days`) + frontend-блок чек-листа.
**Оценка:** M-L (новый endpoint + компонент; зависит от D1 для measurement-критериев, но ROM/боль-критерии можно вывести сразу).

**D3. Поддержка не-хирургических anchor + Фаза 0 + бессрочные/откатные фазы.** `program_types.anchor` (микро-миграция) + логика «Путь» не показывать «неделя от операции» для criterion_driven + корректный рендер финальной/бессрочной фазы + (опц.) регресс фазы.
**Оценка:** M.

**Порядок внутри D:** D2 (ROM/боль-критерии) → D3 → D1 (тяжёлое measurement-хранилище) — чтобы быстро дать ценность на том, что уже хранится.

## 6. Открытые клинические вопросы для эксперта (Vadim)

1. **Сколько program_types на старте?** Дробить ПКС и ЗКС на хирург+консерватив (2+2) сразу или начать только с хирургических? Дробить хрящ на 6 (процедура×локализация) или 1 + флаг?
2. **Силовые/hop LSI ≥90% — главный RTS-гейт во многих досье, но валидирован только для ACL.** Для PCL, EMR, тендинопатии, хряща пороги **заимствованы из ACL** и помечены «не валидировано». Принимаем 90% как дефолт-ориентир везде или калибруем по патологии?
3. **ACL-RSI — время-зависимый порог** (56→65→72→76 при 4/6/12/14 мес). Схема хранит ОДИН `threshold_value` на критерий. Берём фиксированный 72 (12 мес) или нужен механизм меняющегося порога?
4. **24-часовая реактивность боли** (ITBS, PFPS, тендинопатия, OA) — как операционализировать? Порог ≤2/10 vs ≤3/10 (PFPS), до 5/10 (Silbernagel/OA). Это паттерн из ≥2 замеров — это measurement или self_report?
5. **`effusion_grade` (0-3) vs существующий `swelling` (0-3) в diary** — мапить на одну шкалу или заводить отдельную?
6. **Какие критерии `is_required = true`?** Досье дают много гейтов; не все обязательны (напр. `posterior_laxity_diff_mm` помечен «опционально»). Нужен экспертный отбор обязательных vs справочных.
7. **Surgeon-overridable числа** (WB/ROM/брейс/сроки в хирургических). Засеваем дефолтные числа из досье как ориентир, или оставляем пустыми пока хирург не введёт? Нужен ли per-patient override (D-работа)?
8. **Расхождение сроков RTS** (ACL: 9 мес vs Kotsifaki «время не связано с риском у выполнивших критерии»). Жёсткий time-гейт ≥9 мес ставим или мягкий?
9. **knee_general (пустой)** — оставляем как «прочее колено без протокола» или замещаем `knee_oa`/конкретными?
10. **PROMs-модуль** (IKDC, KOOS, VISA-P, Lysholm, ACL-RSI) — нужен ли вообще на пилоте, или пока эти критерии = instructor_check?

## 7. Рекомендованная последовательность

**Цель: минимальный ценный кусок → пилот, не ломая живой прод-ACL.**

**Шаг 0 (решения, блокирует всё):** §6 вопросы 1, 2, 5, 6 закрыть с Vadim. Особенно — **нормализация словаря measurement_type** (свести синонимы, утвердить LSI-неймспейс). Без этого Слот C расползётся.

**Шаг 1 — knee_oa как первый новый протокол (закрывает пустоту `knee_general`).**
- Почему первым: чистый консерватив, **нет** хирург-overridable чисел, **нет** зависимости от D1 (силовых/hop хранилищ — у OA числовые гейты слабые: KOOS-баллы, 30s-CST reps, вес%, боль≤5/10). Гейты в основном instructor_check/self_report → засеваются и работают сразу.
- Слоты: A (1 program_type) + B (4 фазы) + C (10 критериев). Минимум D.
- Даёт первому неоперированному пациенту непустой «Путь».

**Шаг 2 — расширить эталон ACL (Фаза 0 prehab + Фаза 6 профилактика) на `acl`.**
- Аккуратный additive seed поверх живых 6 фаз. Проверяет, что seed-механика не ломает прод. Низкий риск (ACL уже работает).

**Шаг 3 — D2 (вывод авто-критериев в «Путь») на том, что УЖЕ хранится (ROM-градусы + боль-ВАШ).**
- Даёт видимую ценность: галочки «сгибание ≥90° ✓» в «Пути» по реальным замерам Волны 2. Работает для ACL/PCL/EMR/TKA/остеотомии (ROM-гейты) сразу.
- + D3 (anchor-признак, чтобы criterion_driven протоколы не врали про «недели от операции»).

**Шаг 4 — добить хирургические протоколы seed'ом (A+B+C):** TKA, EMR, остеотомия, хрящ (по основной траектории, без дробления на 6), PCL postop. ~их фазы/критерии ложатся в существующую схему; силовые/hop-гейты пока instructor_check.

**Шаг 5 — добить консервативные:** ITBS, PFPS, тендинопатия. Здесь же закрыть двунаправленные переходы/бессрочные фазы (D3 расширение).

**Отложить (после пилота):**
- **D1** (новое measurement-хранилище для силы LSI%/hop%/баланс/PROMs) — самая дорогая работа (≈Волна 2). До неё ~30 критериев живут как instructor_check — клинически приемлемо (инструктор и так меряет очно).
- Дробление хряща на 6 program_type.
- Консервативные ПКС/ЗКС (2 доп. program_type) — если пилот без copers.
- Per-patient surgeon-override порогов.
- Время-зависимый ACL-RSI порог.

**Логика приоритета:** сначала то, что **не требует кода** (seed A/B/C на existing-схеме) и **не зависит от нового хранилища**; среди патологий — сначала консервативная OA (нет хирург-чисел), потом эталон-ACL расширение (низкий риск), потом D2-вывод на existing-замерах (видимая ценность), и только в конце — дорогой D1.
