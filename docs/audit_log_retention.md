# Audit log retention — политика и план активации

**Дата решения:** 2026-04-29
**Статус:** решение принято, имплементация отложена до триггера.

## Контекст

Таблица `audit_logs` пишется на каждое чувствительное действие (READ пациента, CREATE/UPDATE/DELETE медзаписей, OAUTH_LOGIN, etc). Без ротации растёт бесконечно.

## Текущий объём

На 2026-04-29 в dev БД:
- 19 записей за 2 дня = ~10 записей/день при 1 user'е
- Размер таблицы: 168 KB (включая JSONB metadata)

## Прогноз по масштабу

| Сценарий | Записей/день | Записей/год | Размер таблицы/год |
|---|---|---|---|
| Текущий dev (1 user) | ~10 | ~3.7k | ~1 MB |
| Пилот (10 пациентов × 30 действий) | ~300 | ~110k | ~10 MB |
| 100 пациентов | ~3000 | ~1.1M | ~100 MB |
| 1000 пациентов | ~30000 | ~11M | ~1 GB |

PostgreSQL без проблем тянет таблицы 100M+ строк при правильных индексах. Реальная проблема возникнет ближе к 10M записей или когда GDPR/152-ФЗ запросы потребуют быстрого поиска по audit_logs (через индекс на patient_id уже есть, через created_at тоже добавим при активации).

## Юридический контекст

**ФЗ-152 «О персональных данных»** не содержит конкретного срока хранения логов доступа — определяется оператором.

**ПП РФ № 1119** требует логи доступа с возможностью идентификации пользователя и времени, но конкретный срок не указан.

**Приказ ФСТЭК № 21** указывает 3 года для государственных ИСПДн — **не применяется** к частной студии физиотерапии.

**Вывод:** строгого требования срока нет. Разумная политика — **2 года** (покрывает любые претензии в типичном гражданском процессе).

## Решение

**Tiered подход** — активируем по мере роста объёма, не преждевременно.

### Tier 0: текущий (≤ 100k записей)
- **Действие:** ничего не делаем.
- **Триггер для перехода:** `SELECT COUNT(*) FROM audit_logs > 100000` ИЛИ размер `> 50 MB`.
- Объём настолько мал что любые VACUUM/index-сканы быстры.

### Tier 1: cron DELETE (100k — 1M записей)
- **Действие:** ежедневный cron (04:30 Екб) удаляет записи старше 2 лет:
  ```sql
  DELETE FROM audit_logs WHERE created_at < NOW() - INTERVAL '2 years';
  ```
- **Заготовка скрипта:** `deploy/audit-log-cleanup.sh` (создан, но не активирован в cron).
- **Индекс:** добавить если ещё нет — `CREATE INDEX idx_audit_logs_created_at ON audit_logs(created_at)` для быстрого DELETE.
- **Триггер для перехода в Tier 2:** объём > 1M записей или DELETE начинает занимать > 30 секунд.

### Tier 2: partitioning by month (1M+ записей)
- **Действие:** мигрировать `audit_logs` в partitioned table по `created_at` с месячными партициями.
- **Преимущество:** `DROP PARTITION` мгновенный, не блокирует таблицу.
- **Cron:** ежемесячно в 1-е число — создать партицию для следующего месяца + drop партиции > 24 месяцев назад.
- **Скрипт:** `deploy/audit-log-rotate-partitions.sh` (написать при активации Tier 2).
- **Совместимость:** PostgreSQL ≥ 12, у нас 18 — без проблем.

## Заготовка для Tier 1

Когда настанет время активации:

```bash
# deploy/audit-log-cleanup.sh
#!/bin/bash
set -euo pipefail

set -a
source <(grep -E '^DB_' /opt/azarean-rehab/backend/.env)
set +a

export PGPASSWORD="$DB_PASSWORD"

DELETED=$(psql -h localhost -U "$DB_USER" "$DB_NAME" -tAc \
  "DELETE FROM audit_logs WHERE created_at < NOW() - INTERVAL '2 years' RETURNING 1;" \
  | wc -l)

TOTAL=$(psql -h localhost -U "$DB_USER" "$DB_NAME" -tAc \
  "SELECT COUNT(*) FROM audit_logs;")

echo "[$(date '+%Y-%m-%d %H:%M:%S')] Удалено: $DELETED, осталось: $TOTAL"
```

Cron entry (в `setup.sh`):
```
30 04 * * * root /opt/azarean-rehab/deploy/audit-log-cleanup.sh >> /var/log/azarean-rehab-audit-cleanup.log 2>&1
```

## Что делать сейчас

**Ничего не активируем.** Объём 19 записей × 2 дня → даже если запустится пилот на 10 пациентов завтра, до 100k записей пройдёт ~333 дня.

**Триггер мониторинга:** добавить в [BACKLOG_PLAN_2026-04-29.md](BACKLOG_PLAN_2026-04-29.md) или в Sentry alerts (когда настроим) — раз в неделю проверять `SELECT COUNT(*) FROM audit_logs`. При приближении к 50k — создать заготовку Tier 1.
